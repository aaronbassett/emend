# Phase 2 Retrospective: Foundational (FFI boundary, core substrate, app shell)

## What Worked Well
- Verifying UniFFI 0.31.1 facts against the actual `mozilla/uniffi-rs` source (not memory) prevented several traps: assuming a `build.rs`/build-dep was needed (it isn't for proc-macro mode), assuming `setup_scaffolding!` needs a namespace arg (optional), and assuming a rich `uniffi::Error` needs an extra `uniffi::Enum` derive to be a by-value parameter (it doesn't).
- Test-first for the Rust core paid off: the document substrate's surrogate-pair/CRLF tests and the fs crash-equivalence/concurrent-read tests caught real edge cases, and one agent's tests caught a genuine `panic_message` `Box`-deref bug.
- Sequencing the emend-core substrate agents (rather than running them in parallel) avoided build races on shared crate compilation / `lib.rs`.
- Empirical bindgen exploration (generate into a temp dir, inspect the actual files) settled the xcframework layout instead of guessing.

## What Didn't Work
- `-default-isolation nonisolated` (research §A1/§C9) does not exist in Swift 6.0.3 — it's a 6.2 flag. The real Swift-6 problem was strict-concurrency on the generated foreign-trait callback **VTable statics**, fixed by compiling the generated `EmendCoreFFI` target in **Swift 5 language mode**.
- `git rm` pre-stages the deletion, which then got swept into an unrelated `git add … && git commit`. Stage deletions deliberately, or check `git status` before committing a curated subset.

## Workarounds & Solutions
- `uniffi-bindgen-swift` is NOT a standalone crate: `cargo install uniffi --version 0.31.1 --features cli --bin uniffi-bindgen-swift`. It takes a **positional** `OUT_DIR` (not `--out-dir`); use `--module-name emend_ffiFFI --modulemap-filename module.modulemap` so the static-lib xcframework's clang module matches the generated `import emend_ffiFFI`.
- Generated bindings + the xcframework are git-ignored build output, regenerated by `scripts/build-xcframework.sh` (and in CI). Split the `.swift` (→ SwiftPM source target) from the `.h`+`module.modulemap` (→ xcframework Headers).
- Security-scoped bookmarks need a signed, sandboxed app, so headless unit tests use **plain** bookmarks via parameterised options; `EmendTests` is app-hosted so it can `@testable import Emend`.

## Packages & Dependencies
- `criterion` pinned to 0.7 (0.8 needs Rust 1.86 > MSRV 1.85). Added a CI `cargo +1.85 check --all` step so MSRV regressions fail mechanically.
- `ropey` 1.6.1 (the 2.0 line is beta), `default-features = false` + `simd`, dropping `unicode_lines`/`cr_lines` so only LF/CRLF are line breaks (TextKit semantics).
- `tempfile` (atomic writes) — std's `File::sync_all()` already issues `F_FULLFSYNC` on Apple targets, so no `libc`/`rustix` fcntl needed.
- `tokio`/`tokio-util` live in `emend-ffi` (boundary async wiring), never `emend-core`.

## Patterns & Code
- **Hard rule: `emend-core` has zero `uniffi`/`tokio` deps** (Constitution V) — verified with `cargo tree -p emend-core -i uniffi`. The FFI error is a separate `FfiError` enum in `emend-ffi` with an **exhaustive** `From<emend_core::EmendError>` (no wildcard arm → adding a core variant fails to compile until mirrored).
- Streaming: foreign-trait sinks (`with_foreign`) bridged to Swift `AsyncStream`/`AsyncThrowingStream`; exactly one terminal per stream; cancellation via the Rust-owned `CancellationHandle.cancel()` wired to the stream's `onTermination`.
- u32 (FFI/UTF-16) ↔ usize (ropey/char) conversions are always checked (`try_from`), never `as`; surrogate-pair splits are rejected, edits are atomic-on-failure.

## For Next Time
- Future perf benches go in `emend-bench` with `std::hint::black_box` (criterion's is deprecated) and must obey the no-panic/unwrap/expect lints (`--all-targets` compiles them).
- When adding any FFI export that streams or cancels, reuse `handles::CancellationHandle` + `runtime()` and wrap spawned work in `panic::contain_panic` so a panic becomes one error terminal, not a process abort.
- Keep a `1.85.0` toolchain installed; run `cargo +1.85 check --all` before adding deps.
