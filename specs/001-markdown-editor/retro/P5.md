# Phase 5 Retrospective: US3 — Quick Open

## What Worked Well
<!-- Effective patterns, tools, approaches -->

- **Parallel split: background `rust-dev` for the FFI seeding slice, Swift overlay
  by hand.** The `reindex_all` FFI gap-fill went to one (and only one) background
  Rust agent while I built `QuickOpenModel`/`QuickOpenView` + the ⌘P wiring against
  the already-shipped search bindings — no idle wall-clock, no build race. I
  re-ran the full Rust gate (fmt/clippy/test/MSRV + core uniffi/tokio-free 101/101)
  myself before committing the agent's work.
- **US3 was mostly composition of existing pieces.** The streaming driver (T073)
  reused the foundational `SearchSink`/`SearchHandle`/`CancellationToken`; the
  seeding method reused US2's `collect_files` + `rel_for` + `Index::rebuild`. Net
  new core logic ≈ zero, new deps = 0.
- **The `FsObserver` Sendable-bridge pattern transferred verbatim** to
  `QuickOpenSink` (a `final class` holding immutable `@Sendable` closures that hop
  to `@MainActor` via `Task`). Swift 6 strict-concurrency clean on the first try.
- **Headless, model-driven test proved the full async streaming path
  deterministically** — drive `QuickOpenModel` + a real `WorkspaceHandle`, await a
  runloop spin for the streamed batch, assert open-flow — no XCUITest needed.

## What Didn't Work
<!-- Problems, issues, limitations encountered -->

- **The index was never seeded in the running app — a latent US2 gap US3
  surfaced.** `add_location`/`list_children` don't populate the index; only
  `rebuild_index` did, and *nothing called it*. US2's tests seeded the index
  directly, so the gap stayed invisible until US3 needed live results. Closed with
  `WorkspaceHandle::reindex_all` + `WorkspaceModel` wiring (launch/add/remove
  location, off-main).
- **External-change incremental index maintenance is still missing (FR-017a,
  partial).** Internal file ops (create/rename/move/delete *through the app*)
  update the index incrementally, but a watcher-driven *external* change refreshes
  the sidebar without touching the core index — so a file another tool creates or
  deletes won't appear/disappear in Quick Open until the next full reindex
  (launch / add / remove location). A full reindex on every external event would
  violate FR-017a's "incremental, no full rescan" clause, so the real fix needs
  new FFI index-mutation methods (`index_insert`/`index_remove`/`index_rename`)
  wired to the watcher's `DocObserver`. **Deferred to a follow-up; noted in the PR.**
- **Folder-in-results deferred.** FR-017 lists files *and* folders; `collect_files`
  returns files only and results open into editor tabs (only files are openable),
  so v1 seeds + opens files. Folder navigation needs sidebar-reveal infra that
  doesn't exist yet — deferred (the testable SC-004 / acceptance test is
  file-centric).

## Workarounds & Solutions
<!-- Bug fixes, non-obvious solutions, clever workarounds -->

- **Canonicalization parity for seeding.** `collect_files` canonicalizes paths
  (`/var` → `/private/var` on macOS); the *location root* must be canonical too, or
  `rel_for` can't prefix-match and the breadcrumb falls back to the basename. Both
  the Rust unit test and the Swift headless test canonicalize the temp root
  **before** `add_location`.
- **Supersede double-guard (NFR-002).** The FFI already cancels the prior in-flight
  query on each `quick_open_query`; the Swift model adds a **monotonic
  `generation`** so a late batch from a just-superseded query (in flight when the
  cancel landed) is dropped when it hops to the main actor, instead of polluting
  the new query's list.
- **Off-main seeding (NFR-001).** `reindex_all` is synchronous (walk + rebuild);
  `WorkspaceModel.reindexInBackground` runs it on `Task.detached(.utility)` so the
  launch / add-location path never stalls the UI.

## Packages & Dependencies
<!-- Useful packages discovered, packages to avoid, version issues -->

- **No new dependencies in US3.** The streaming driver used the full `nucleo` 0.5
  already in the catalog (its own threadpool — keeps `emend-core` tokio-free);
  seeding reused US2's `collect_files`. MSRV 1.85 unaffected (verified
  `cargo +1.85.0 check --all`).

## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->

- **Sendable foreign-sink bridge** (`QuickOpenSink`) = the `FsObserver` pattern: a
  `final class` conforming to the UniFFI foreign trait, holding only immutable
  `@Sendable` closures, hopping each callback to `@MainActor` via `Task`.
- **`attach(workspace:onOpen:)` post-init wiring** (like `ConflictController`) for a
  `@StateObject` view-model that depends on sibling `@StateObject`s the initializer
  can't see; called from MainWindow's `.task`.
- **⌘P shortcut** = a hidden `Button(...).keyboardShortcut("p", modifiers: .command)`
  in `.background` — registers a window-wide shortcut with no toolbar item.
- **Palette key handling** = `.onKeyPress(.upArrow/.downArrow/.escape)` on the
  container (macOS 14+) for selection nav while the `TextField` keeps focus;
  Return via `.onSubmit`.
- **Full-rebuild-on-rare-event indexing**: a complete `reindex_all` on
  launch/add/remove-location is acceptable (rare, fast, off-main); per-edit changes
  stay incremental through the file-op methods. Don't reindex on every keystroke or
  every fs event.

## For Next Time
<!-- Improvement suggestions, lessons learned, things to try differently -->

- **Wire derived state end-to-end inside the slice that builds it.** US3 hit *two*
  US2 gaps (no seeding trigger; no external-change index maintenance) because US2
  built the index plumbing but never connected it to the live app or the watcher. A
  "prove the index reflects a real on-disk change" task in US2 would have caught
  both.
- **The external-change → incremental-index follow-up** (`index_insert`/`_remove`/
  `_rename` FFI + watcher wiring) should land **before US5** — wiki-link resolution
  (US5) leans on the same index (`resolve_name`), so a stale index hurts there too.
- Carried from US1/US2: XCUITest can't bootstrap under CI's `CODE_SIGNING_ALLOWED=NO`;
  keep `[test]` UI flows as headless app-hosted tests driving the view-model + real
  handles (now also `QuickOpenTests`).
- Carried from US2: surface FFI gaps (icon/favorite/pin getters, core-owned
  app-state persistence) — still open; consider a small FFI follow-up before US6.
