# Phase 4 Retrospective: US2 — File-based workspace

## What Worked Well
<!-- Effective patterns, tools, approaches -->

- **Sequenced background `rust-dev` agents for the core slices.** workspace → index →
  watcher/conflict → FFI, one agent at a time (never two Rust agents concurrently,
  to avoid build races). Each returned a structured data report + a green gate; I
  re-ran the gate myself before committing. A transient API-death mid-watcher
  recovered cleanly (kept the agent's verified `notify` deps, re-dispatched the rest).
- **Model/view split on the Swift side** kept the intricate AppKit testable-where-it-matters:
  `WorkspaceModel`/`TabModel` (logic, headless-tested) vs `WorkspaceOutlineView`/cells
  (pragmatic-UI glue). The headless `WorkspaceFlowTests` drives the real
  `WorkspaceHandle` + models for the whole add→tree→open→move flow.
- **Targeted `reloadItem` + a stable Favorites group node** gave live refresh and
  favorites without collapsing expanded folders (a revision tick for structural
  changes; a pending-reloads list for fs-driven folder refresh).

## What Didn't Work
<!-- Problems, issues, limitations encountered -->

- **The FFI surface had gaps the data-model implied:** no getter for folder-icon /
  isFavorite / isPinned, and the core holds favorites/pins/icons **in memory only**
  (no persistence layer, despite the data-model's "core-owned app-support store").
  Result: Swift became the display source of truth + persistence (UserDefaults),
  forwarding to the core via setters and replaying on launch. A future refactor
  should move this core-side (tracked in CONCERNS.md RUNTIME-001/002).
- **The live-refresh + conflict runtime path is not automatically verifiable** — it
  depends on real OS filesystem events, which the headless suite can't exercise. The
  pure watcher logic is unit-tested in Rust, but the Swift wiring (DocObserver bridge,
  reload, banner, self-write timing) needs **manual UI verification**.
- **A second Bash-classifier outage** stalled the loop for ~3 ticks mid-T062; recovered
  by retrying with backoff (work stayed uncommitted-but-safe in the tree).

## Workarounds & Solutions
<!-- Bug fixes, non-obvious solutions, clever workarounds -->

- **SwiftLint size limits as files grew:** `type_body_length` (250) → move private
  helpers into a same-file `private extension`; `file_length` (400) → split a type out
  (`WorkspaceNode.swift`). Both keep diffs clean and don't fight SwiftFormat.
- **The SwiftFormat/SwiftLint `opening_brace` conflict recurred** on multi-line
  `if`/`guard` conditions — precompute a boolean (`let x = …; if x {`) so the brace
  stays on the condition line.
- **Self-write suppression made robust** by NOT relying solely on cross-language
  (mtime,len) identity matching: the authoritative suppressor is a Swift-side
  per-path time window in `ConflictController`; the core's `recordSelfWrite` is
  best-effort on top.

## Packages & Dependencies
<!-- Useful packages discovered, packages to avoid, version issues -->

- Carried from US1: pin every new dep to `rust-version ≤ 1.85` (CI runs `cargo +1.85 check --all`); verify on crates.io first. US2 wires the first watcher/index deps — `notify` + `notify-debouncer-full` (T057) and `nucleo` (T058). Confirm their MSRV before pinning.
- **`nucleo-matcher 0.3.1`** (not full `nucleo`) for the synchronous in-memory index —
  no threadpool/async, keeps `emend-core` tokio-free. No declared `rust-version`;
  MSRV verified empirically via `cargo +1.85 check`.
- **`notify 8.2` + `notify-debouncer-full 0.7`** (notify 8.x; 9.0 is RC). notify uses
  its own OS threads (not tokio) — posts to a `std::sync::mpsc` channel.

## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->

- **NSOutlineView in SwiftUI:** `NSViewRepresentable` + a `@MainActor` Coordinator
  conforming to `NSOutlineViewDataSource`/`Delegate`/`NSMenuDelegate`. Those SDK
  protocols ARE `@MainActor`, so plain conformance works (unlike
  `NSTextStorageDelegate`, which needed `nonisolated` + `assumeIsolated`). Items are
  reference-type nodes (object identity); children load lazily and cache.
- **Cross-thread Rust→Swift callbacks:** a `final class` conforming to the uniffi
  foreign trait (`DocObserver`), holding only an immutable `@Sendable` closure (so it's
  `Sendable`), that hops to the main actor via `Task { @MainActor in … }`.
- **Editor recreation on reload:** include a per-tab `reloadToken` in the SwiftUI
  `.id(...)` so reopening a document tears down and rebuilds the live editor with
  fresh on-disk text.
- **App-state persistence (favorites/pins/icons):** Swift-side `Codable` `AppState` in
  `UserDefaults`, forwarded to the core via setters and replayed on launch (the FFI
  has no getters and the core doesn't persist). Path-keyed; repathed on move.

## For Next Time
<!-- Improvement suggestions, lessons learned, things to try differently -->

- Carried from US1: T067's `WorkspaceFlowTests` is specced as XCUITest, but the XCUITest runner can't bootstrap under CI's `CODE_SIGNING_ALLOWED=NO` (CI is GUI/signing-free by design). Implement it as a **headless app-hosted integration test** (drive the workspace model / outline data source directly), mirroring US1's `EditorPersistenceTests` decision.
- **Surface FFI gaps to the core slice early.** Had `WorkspaceHandle` exported icon/
  favorite/pin getters + a persistence load/save, the Swift side would be simpler and
  the data-model honored. Consider a small follow-up FFI task before US6.
- **Flag manual-verification needs explicitly** in the PR — the live-refresh/conflict
  UI can't be proven by CI, so a reviewer should drive it by hand.
