# Phase 9 Retrospective: US7 — Typography & appearance

Scope: curated typography with user customization (font / size / line + paragraph
spacing) applied to BOTH the editor and the preview, and following the native
light/dark appearance automatically (FR-038/FR-039).

## What Worked Well
<!-- Effective patterns, tools, approaches -->

- **One `Typography` resolver, two appliers** kept the editor and preview in sync from
  a single source: `Typography.font`/`paragraphStyle` for the `NSTextView`,
  `Typography.previewCSS` injected after `theme.css`. Clamping lives once in the core
  (`set_typography`), so the UI can't produce a broken layout regardless of caller.
- `TypographySettings` is a uniffi `Record` and **`Equatable`**, so the editor's
  `updateNSView` change-guard (`if typography != coordinator.typography`) and the
  preview's `didSet` version-bump were trivial — no manual field diffing.
- Live apply was cheap to bolt on: the editor already re-attributes the whole document
  each edit, so `applyTypography` just resets the font/paragraph style and re-attributes;
  the preview just re-injects CSS over the unchanged HTML via a `version` bump.

## What Didn't Work
<!-- Problems, issues, limitations encountered -->

- Making `PreviewModel.themeCSS` `private` (after splitting out the combined `css`)
  broke `exportPDF`, which had read `themeCSS` directly — a quiet compile break caught
  by the build, not the design.

## Workarounds & Solutions
<!-- Bug fixes, non-obvious solutions, clever workarounds -->

- Exposed `PreviewModel.css` (syntect theme **+** live typography) as the single public
  stylesheet; `exportPDF` now uses it too, so the **PDF matches the on-screen preview's
  typography** for free.
- A typography change bumps the preview's `@Published version` (HTML unchanged), which
  re-triggers `updateNSView` → `__emendRender` re-injects the new CSS — no re-render of
  the document needed.

## Packages & Dependencies
<!-- Useful packages discovered, packages to avoid, version issues -->

- No new deps expected. `emend-core` stays tokio/uniffi/reqwest-free; MSRV 1.85.

## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->

- **Core holds `TypographySettings` in-memory with defaults + clamping**; the FFI
  exposes `get_typography`/`set_typography` (contract §8). Per the established US2
  app-state pattern, **persistence is Swift-side** (UserDefaults) and replayed into
  the core on launch — the data-model envisions a core-owned app-support store, but
  the core still has no persistence layer.
- Typography applies to **two render surfaces**: the editor (`NSTextView` base font +
  line spacing via paragraph style / `defaultParagraphStyle`, re-attributed) and the
  preview (CSS variables injected into the WKWebView — extend the `bridge.js`
  theme-injection used for the syntect CSS). One settings source, two appliers.
- **Light/dark follows the system automatically** (SwiftUI/AppKit `NSAppearance`); the
  preview CSS already uses `prefers-color-scheme`. v1 has no manual theme toggle — so
  `TypographySettings` carries font/size/spacing, not a theme field.
