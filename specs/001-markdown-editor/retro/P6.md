# Phase 6 Retrospective: US4 — Faithful preview + PDF export

## What Worked Well
<!-- Effective patterns, tools, approaches -->

- **comrak `render.sourcepos` → `data-line` anchors** are a clean, stable basis for
  scroll sync. A post-pass on the rendered HTML inserts `data-line="<startLine>"`
  beside each `data-sourcepos`; bridge.js turns those into an offset table. Editor and
  preview both key on 1-based source lines, so the two directions stay symmetric.
- **One offline `WKWebView` shell, two render entry points.** The live preview uses a
  fire-and-forget `__emendRender`; PDF export uses `__emendRenderForPrint` (async, awaits
  Mermaid). `renderMermaid` returns the `mermaid.run` promise so the print path can await
  async diagram layout while the live path ignores it. No duplicated render logic.
- **Headless WKWebView print *is* testable** under the app-hosted `EmendTests` host on a
  machine/runner with a window server — once the deadlock (below) was fixed, the
  multi-page assertion runs in ~0.2 s. No XCUITest needed.
- The TextKit 2 geometry for scroll sync (`characterIndexForInsertion(at:)` forward,
  `textLayoutFragment(for: NSTextLocation)` + `textContainerOrigin` reverse) maps
  top-visible-line ↔ y-offset without dropping to TextKit 1.

## What Didn't Work
<!-- Problems, issues, limitations encountered -->

- **`NSPrintOperation.run()` (synchronous) deadlocks** an off-screen `WKWebView` print
  when called from a `@MainActor async` context. It blocks the main thread/run loop while
  WebKit's print path needs that same run loop to deliver web-content-process IPC →
  hang forever. First full test run hung ~25 min before I killed it. Diagnosed with
  `NSLog` checkpoints under a hard shell `timeout`: the log reached "running print
  operation" and never returned.
- swiftformat wraps a long inline `withCheckedThrowingContinuation { (c: CheckedContinuation<Bool, Error>) in`
  generic across three lines (ugly but lint-clean). Extracting per-step helper methods
  (`loadTemplate`/`renderContent`/`paginate`) both fixed `function_body_length` and kept
  the generic on one line per helper.

## Workarounds & Solutions
<!-- Bug fixes, non-obvious solutions, clever workarounds -->

- **PDF export uses the asynchronous `NSPrintOperation.runModal(for:delegate:didRun:contextInfo:)`**
  (with `showsPrintPanel=false`, save disposition) instead of `run()`. It returns
  immediately and fires `printOperationDidRun(_:success:contextInfo:)` when the file is
  written; a continuation bridges that to `async`. No main-thread block → no deadlock.
- Both async WebKit steps (load, print) have a `DispatchQueue.main.asyncAfter` **watchdog**
  that resumes the continuation with an error if WebKit stalls, so an export can never hang
  the app (or a test) indefinitely.
- The off-screen print host keeps the `WKWebView` in a borderless, far-off-screen
  (`-10000,-10000`) `NSWindow` ordered front-regardless, so WebKit lays out and runs
  Mermaid's async JS rather than throttling an occluded view.
- Editor scroll observation uses a **selector-based** `NSView.boundsDidChangeNotification`
  observer (not the block form): a `@objc` method on the `@MainActor` `EditorCoordinator`
  sidesteps the Swift 6 `@Sendable`-closure non-Sendable-`self` capture problem.

## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->

- **Privacy (SC-008/FR-035) is binding for the preview:** the `WKWebView` must block all
  remote loads — offline CSP + `nonPersistent` data store + a navigation-blocking delegate.
  Mermaid + KaTeX are **bundled** (vendored under `app/Emend/Emend/Resources/preview`, already
  a folder reference in `project.yml`); `template.html` resolves them via relative paths, so
  point the WebView `baseURL` at that folder. Zero outbound network from preview, ever.
- Scroll sync (T088) uses `data-line` anchors emitted by comrak (T084); a short per-side
  mute (`muteEditor`/`mutePreview`, ~160 ms) guards the bidirectional feedback loop, and a
  JS `applyingScroll` flag swallows the scroll events our own `scrollTo` causes.
- PDF export (T089) via the **async** `NSPrintOperation.runModal(for:…)` on an offscreen
  `WKWebView` with `@media print`/`@page` rules in `theme.css` — never the synchronous
  `run()` (deadlocks; see above).
- `ScrollSync` is a shared `@StateObject` hub in `MainWindow`; the **active** editor (by
  `isActive: tab.id == activeID`) registers its scroll view, so the multi-editor ZStack
  drives sync only from the visible tab.
